# Next.js Stack Developer Guide

Welcome to the `nextjs` stack implementation for the **Fluksos CLI**. 

This document is for **developers and contributors** who are maintaining, modifying, or extending the Next.js stack generator. If you are an end-user of the CLI looking for usage instructions, please refer to the [root README](../../README.md).

---

## ­¤ÅŚ’ĖÅ Architecture of the Generator

The Next.js generator is driven by a deterministic pipeline executed by `scripts/init_project.js`. The pipeline is strictly ordered and guarantees that the resulting project is ready for production immediately after scaffold.

### The Pipeline Steps (`init_project.js`)

1. **Assert Preconditions**: Validates that `pnpm` is available globally.
2. **Assert Required Template Files**: Fails early if any template file or folder is missing, preventing partial or broken scaffolds.
3. **Create Workspace Root**: Generates the root `package.json`, `turbo.json`, and `pnpm-workspace.yaml`.
4. **Scaffold Next App**: Runs `pnpm dlx create-next-app` inside `apps/web/` with strict flags (`--turbopack`, `--no-eslint`, `--skip-install`). Immediately cleans up default lockfiles.
5. **Cleanup Generated Files**: Removes default configurations (ESLint, Prettier, default Tailwind configs, Babel) generated by Next.js.
6. **Install Common Dependencies**: Installs workspace-level (`turbo`, `@biomejs/biome`, `lefthook`) and common app-level dependencies (Next.js, React, Tailwind, Biome, Valibot, Zustand, Otel).
7. **Install Tier Dependencies**: Installs specific tools depending on the chosen `--tier` (e.g., `next-safe-action` for Tier 2; `drizzle-orm`, `better-auth`, `@tanstack/react-query` for Tier 3).
8. **Apply Templates (Root)**: Copies `templates/root/` configuration files.
9. **Apply Templates (Common)**: Copies `templates/app-common/` directly into `apps/web/`.
10. **Apply Templates (Tiers)**: Overlays `app-tier-2/` and `app-tier-3/` as requested.
11. **Apply Observability**: Adds Docker Compose and K6 test templates.
12. **Setup Lefthook**: Initializes Git and installs `lefthook` for pre-commit boundaries.
13. **Validate Scaffold**: Asserts that all invariants (required files) are present and anti-invariants (banned files) are absent.

> **CRITICAL**: Dependency versions are strictly pinned in the `STACK_VERSIONS` object at the top of `init_project.js`. Do not update versions directly inline.

## ­¤ōü Template Structure

Templates are **static files**, not code generators. What you put in `templates/` is verbatim what the user receives. There are no placeholder strings (e.g., `{{ projectName }}`) inside these files.

| Directory | Purpose |
|-----------|---------|
| `root/` | Workspace-level tools (`biome.json`, `lefthook.yml`) and Enterprise CI/CD pipelines (`.github/workflows/ci.yml`). |
| `app-common/` | Base Next.js files applied to **all** tiers. Includes `globals.css`, `next.config.ts`, `tsconfig.json`, `components.json`, and basic AEO routing (`llms.txt`, `robots.ts`, `sitemap.ts`, `public/mirrors/home.md`). |
| `app-tier-2/` | Applied if Tier >= 2. Overlays `lib/safe-action.ts` and `lib/fetcher.server.ts`. |
| `app-tier-3/` | Applied if Tier === 3. Overlays `db/index.ts`, `drizzle.config.ts`, `lib/auth.ts`, and the Better Auth API route. |
| `observability/`| Docker Compose stack for OpenTelemetry (Prometheus, Tempo, Loki, Pyroscope, Grafana). |
| `tests/` | K6 performance test scripts. |

## ­¤øĀ’ĖÅ Modifying Generators

The `generate_action.js` and `generate_rpc_hook.js` scripts are AST-blind templates. They do not parse AST to inject code; they generate new files based on string replacement.

When modifying generators:
1. Ensure the output strictly adheres to the AST rules (e.g., using `valibot` for schemas, avoiding `fetch` inside components).
2. For Server Actions, ensure the generator wraps the action with `actionClient` so the Upstash Rate Limiter (implemented in Tier 2/3) is applied automatically.
3. For RPC Hooks, ensure the generator uses `@tanstack/react-query` to wrap Hono, preventing direct client calls.

## ­¤øĪ’ĖÅ Modifying Validators and AST Rules

The Next.js stack enforces strict architectural rules via independent validator scripts. These are used during the scaffolding validation phase and can be run by the user via `npx fluksos validate nextjs <dir>`.

### 1. Architectural Validation (`validate-architecture.js` + `lib/ast-parser.js`)
This is the core engine ensuring the framework is used correctly. 
- **Tooling Integrity**: Bans ESLint (`.eslintrc.*`) in favor of Biome.
- **Routing**: Bans `src/pages/` (Forces App Router). Bans Next.js `middleware.ts` in favor of Edge-compatible `proxy.ts`.
- **AEO / SEO Rules**: Enforces the presence of `llms.txt` and `llms-full.txt`. Forbids static `robots.txt` and `sitemap.xml` (forces dynamic `.ts` equivalents).
- **Tailwind**: Forbids `tailwind.config.ts` (enforcing Tailwind v4 via `globals.css`) and enforces `.mjs` for PostCSS.
- **Marketing Graph**: Requires an `opengraph-image` alongside any public marketing `page.tsx`.
- **AST Parsing Rules**:
  - `page.tsx` must be a React Server Component (no `'use client'`).
  - No direct `db` imports in `page.tsx` or Client Components (enforced via `db-in-controller`).
  - `MD5` is fully banned (`banned-md5`).
  - Warning on `uuidv4()` if `uuidv7()` should be used (`legacy-uuid`).
  - Server actions, `db`, and services must include `import 'server-only'`.
  - Global state must use Zustand (React `createContext` and Redux are banned).
  - Client data fetching must use TanStack Query (no raw `fetch()` or `axios()` inside `useEffect`).
  - Zustand stores cannot fetch remote data (pure UI state only).
  - No `@ts-ignore`. No `axios`. No `@import url()` in CSS.
  - Images must have `alt` text.

### 2. Configuration Validation (`validate-config.js`)
Validates structural correctness of config files:
- **package.json**: Scans for and bans forbidden dependencies (e.g., `axios`, `jest`, `next-auth`, `zod`, `moment`, `husky`). Enforces `check` and `lint:fix` Biome scripts. Detects insecure `db:push` scripts.
- **tsconfig.json**: Enforces strictness (`noImplicitAny` must not be false).
- **next.config.ts**: Verifies that `experimental: { typedRoutes: true }` is enabled.
- **components.json**: Asserts Tailwind v4 compatibility for Shadcn (`"config": ""`).
- **.env**: Detects secrets leaked to the browser via the `NEXT_PUBLIC_` prefix.
- **Lockfiles**: Forbids `package-lock.json` and `yarn.lock` (enforces `pnpm`).

### 3. UI State Boundary Validation (`validate-ui-state.js`)
Specialized scanner for UI boundaries:
- Enforces that Zustand stores (`src/stores/`) **must not** contain remote data fetching logic (bans `fetch`, `axios`, `useQuery`).

## ­¤¦¬ Testing

The Next.js stack validates its own rules using a negative-testing approach. 

### The Mock Project
Tests rely on the `tests/mock-project/` directory. This is **not a valid scaffold output**. It is an intentionally broken Next.js project designed to trigger every single rule in our validators.

**Do not fix the code in `mock-project/`**. The tests (`validate-*.test.js`) execute the validators against this directory and assert that the expected `FATAL` and `ERROR` violations are properly caught and reported.

### Running Tests
To run the validator unit tests locally:

```bash
cd stacks/nextjs/scripts
node --test ../tests/validate-architecture.test.js
node --test ../tests/validate-config.test.js
node --test ../tests/validate-ui-state.test.js
```

## ­¤ōØ Modifying the Stack

1. **Adding a Dependency**: Update `STACK_VERSIONS` in `init_project.js` and add the `pnpm add` command in `installCommonDependencies` or `installTierDependencies`.
2. **Adding a Template File**: Place the file in the correct tier directory. Add an `assertFileExists` check in `init_project.js` -> `validateScaffold()`.
3. **Adding a Rule**: 
   - Add the logic to `ast-parser.js` or the relevant validator.
   - Intentionally break the rule inside `tests/mock-project/`.
   - Update the corresponding `.test.js` file to assert the rule is caught.
4. **Structural Changes**: Require an approved `/implementation-plan` via the SpecKit workflow.
